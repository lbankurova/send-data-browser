# Acontext — Lattice/SENDEX Leverage Analysis

**URL:** https://github.com/memodb-io/Acontext
**Verdict:** LOW (with one extractable design idea — see Recommendation)
**One-line:** A self-hosted skill-memory server that auto-distills agent transcripts into Markdown "skill" files; the *storage shape* is what Lattice already has, the *capture pipeline* is the only novel piece, and adopting the library wholesale would impose a 5-container infrastructure tax (Postgres + Redis + RabbitMQ + S3 + FastAPI) for a problem Lattice doesn't actually have.

## What it is (mechanically)

Acontext is an Apache-2.0 "agent skills as a memory layer" system from memodb-io. It treats *memory* and *skills* as the same artifact: every learning ends up as a Markdown file with YAML front-matter (`name`, `version`, `description`, `keywords`) plus prose body. Storage is plain `SKILL.md` files in directories grouped under "learning spaces"; the spaces are exportable as static ZIP archives. There is **no embedding-based retrieval at runtime** — agents access skills through tool calls (`get_skill`, `get_skill_file`, `list_skills`, plus mutation tools `create_skill`, `create_skill_file`, `str_replace_skill_file`, `mv_skill_file`, `delete_skill_file`). The retrieval philosophy is "progressive disclosure": the agent reasons about which skill to pull, rather than a vector search ranking it.

The novel mechanism is the **two-stage learning pipeline**, implemented in `src/server/core/acontext_core/service/skill_learner.py` and the prompts under `acontext_core/llm/prompt/`:
1. **Distillation pass** (single fast LLM call, no lock). Takes a session transcript + outcome label and routes through one of three tools — `skip_learning` (trivial Q&A), `report_success_analysis` (multi-step procedure / debugging / decision process — captures goal, strategy, decisions, generalizable patterns, applicable conditions), or `report_factual_content` (people/preferences/entities, NOT inflated into fake procedures). A separate `report_failure_analysis` path captures task goal / failure point / flawed reasoning / correct approach / prevention principle / applicable conditions.
2. **Skill-Agent pass** (Redis-locked per learning space, queues pending work on lock contention). Reads the existing `SKILL.md` as the "authoritative definition," then *prefers updating an existing skill over creating a new one*: same domain → update; partial overlap → update and broaden scope; zero coverage → only then create at *category* level (e.g., `api-error-handling`, not narrow fixes). Conflict resolution is "consolidation by absorption" — fewer rich skills > many thin ones; no overwriting, no versioning, no `contradicts` edges.

The runtime is a self-hosted multi-service stack: FastAPI core + API layer (`src/server/core` and `src/server/api`), Postgres, Redis, RabbitMQ, S3 (or compatible), plus a Next.js dashboard at :3000. Embeddings exist in code (`acontext_core/llm/embeddings/openai_embedding.py`, `jina_embedding.py`) but the *retrieval* path doesn't use them — they appear to be for session-message dedup or learning-space scoping, not skill ranking. The Claude-Code integration is shipped as a plugin at `.claude-plugin/marketplace.json` → `src/packages/claude-code/plugin/`, which registers an MCP stdio server (`mcp-server.cjs`) and four hook handlers (`SessionStart`, `PostToolUse` matching `*`, `Notification`, `Stop`) — meaning skill capture happens automatically on every tool use and at session end without explicit user opt-in. SDKs ship as `pip install acontext` and `npm i @acontext/acontext`. Bootstrap requires Docker + an OpenAI key.

## Comparison to Lattice's existing memory + knowledge graph

Acontext's schema is strictly weaker than `knowledge-graph.md`. |
| **Index** | `MEMORY.md` hand-curated index with topic groupings | None visible — relies on agent calling `list_skills` and reasoning over names/descriptions | **Lattice** — a curated index gives a human reader the corpus shape; Acontext requires the agent to enumerate every time. |
| **Retrieval at runtime** | Grep across `docs/_internal/` + explicit `Read` with citations | Tool-call (`get_skill`/`get_skill_file`) — agent picks by name/keyword | **Lattice** — Grep returns exact-line citations that the reuse-anchor-drift audit (CLAUDE.md rule 5) and contract-triangle audit (rule 18) depend on. Acontext's tool returns a whole file. |
| **Capture mechanism** | Manual: user types `/lattice:extract-learnings` or directly edits memory files | Automatic: Stop-hook fires on every session end → distillation → skill-agent edits files | **Acontext** — this is the one genuinely novel piece. | **Distillation prompt quality** | `/lattice:extract-learnings` skill prompt + manual prose | Three-way router (`skip_learning` / `success_analysis` / `factual_content`) plus `failure_analysis`, with explicit warnings against "inflating simple factual content into fake procedures" | **Acontext (idea, not the library)** — the success/failure/factual taxonomy and the anti-inflation guardrail are well-thought-out prompt design that Lattice could borrow. |

## Concrete leverage opportunities (if any)

The earlier instructions named four real Lattice gaps. Honest assessment of each against Acontext:

1. **Semantic retrieval across the research corpus (~30 files).** Acontext does **not** solve this — its retrieval is tool-call by name/keyword, not semantic search. Embeddings exist in code but aren't wired to skill retrieval. Even if they were, the "small corpus, grep wins" principle the user named applies. **No leverage.**

2. **Cross-conversation continuity beyond what MEMORY.md indexes.** Acontext's automatic capture-on-Stop-hook is the *one* place it potentially adds something. But Claude Code already loads `MEMORY.md` per session, and the gap is "things the user hasn't yet decided are durable enough to memorialize" — which is exactly what Acontext's distillation tries to automate. **Marginal leverage as an idea (auto-distill on session end), low leverage as a library — Lattice can implement a SessionStop hook that calls `/lattice:extract-learnings` without standing up Postgres + Redis + RabbitMQ + S3.**

3. **Automatic detection of "this memory is now stale because the file it cites moved/deleted."** Acontext does **not** solve this. Its update model broadens scope rather than retiring claims; it has no mechanism to detect that a `file:line` citation no longer resolves. Lattice's `audit-contract-triangles.py` and reuse-anchor baseline already do this for triangle/reuse citations; the gap for *memory* citations remains in both systems. **No leverage.**

4. **Conflict resolution when two memories contradict.** Acontext's answer is "consolidate into one richer skill," which is *strictly weaker* than `knowledge-graph.md`'s `contradicts` edges + audit script. The typed-graph approach catches HCD-FACT-008 ↔ FACT-010 mechanically; Acontext would silently merge them into prose where the disagreement is invisible. **Negative leverage — adopting Acontext's model would regress this property.**

The single extractable idea is the **distillation prompt taxonomy**: the explicit three-way router (`skip_learning` for trivial, `report_success_analysis` for procedural, `report_factual_content` for entities/preferences) plus `report_failure_analysis` with named fields (failure point, flawed reasoning, correct approach, prevention principle, applicable conditions), plus the explicit anti-inflation guardrail ("do NOT inflate simple factual content into fake procedures"). This maps directly onto `/lattice:extract-learnings` and could improve the skill's output quality. The relevant source file is `src/server/core/acontext_core/llm/prompt/skill_distillation.py` (visible via `raw.githubusercontent.com`); the skill-agent prompt `src/server/core/acontext_core/llm/prompt/skill_learner.py` adds the "update over create / broaden scope / fewer-rich-skills > many-thin-ones" consolidation rule, which is also pitchable as a `/lattice:extract-learnings` directive.

## Risks / mismatches

- **Infrastructure tax for a problem we don't have.** Acontext mandates Postgres + Redis + RabbitMQ + S3 + FastAPI + Next.js dashboard for a corpus that, in Lattice's case, is ~50 markdown files. The CLAUDE.md project explicitly notes "if the corpus is small enough that grep wins on latency + transparency, don't add a vector DB." Same logic applies here.
- **OpenAI dependency.** The bootstrap requires an OpenAI API key. Lattice's existing memory has no model dependency — the markdown files don't need an LLM to be read.
- **Schema lock-in regression.** Acontext's `SKILL.md` schema (name/version/description/keywords + prose) is strictly weaker than Lattice's typed `knowledge-graph.md` schema. Migrating typed facts into Acontext skills would lose `contradicts`, `derives_from`, `scope`, `confidence` — all of which the contradiction audits depend on.
- **MCP / hook architecture has security surface.** The Claude-Code plugin auto-installs four hooks, including `PostToolUse` matching `*` (every single tool call). That's a non-trivial intercept point for a third-party plugin in a regulated-data project. SENDEX deals with pre-clinical study data; sending session transcripts to a self-hosted-but-still-distinct-stack process is a data-flow change that would need explicit governance.
- **No staleness model.** The capture-only design means the corpus monotonically grows. Without a retire/retract mechanism, Acontext skill spaces will rot the same way ad-hoc memory files do.
- **"Skill is Memory, Memory is Skill" conflates two things.** Lattice deliberately separates `feedback_*.md` (process learnings — durable but informal) from `knowledge-graph.md` (atomic domain facts — typed and contradictable) from `research/*.md` (corpus of investigated topics). Acontext flattens all three into one shape, losing the distinctions Lattice's rules 18 / 22 are built on.
- **Auto-capture without confirmation is a known anti-pattern.** Lattice's framework explicitly distinguishes *what the user has decided is durable* (MEMORY.md / knowledge-graph.md, hand-promoted) from session ephemera. Auto-distillation on every Stop creates the opposite policy: capture-by-default, edit-out-later. That's the wrong default for a project where the user's tacit signal (deciding to write something down) is itself meaningful.

## Recommendation

**Do not adopt the library or the runtime.** The infrastructure cost (Postgres + Redis + RabbitMQ + S3 + Docker stack) is unjustifiable for a corpus where grep over markdown wins on every axis Lattice cares about: zero cost, exact-line citations, no model dependency, no plaintext-to-third-party concern, and full git-portability. The conflict-resolution model is *strictly worse* than `knowledge-graph.md`'s typed `contradicts` edges. The retrieval API offers nothing semantic — it's tool-call by name, which is what Grep + `MEMORY.md` already provides with better citation precision.

**Extract one design idea: the distillation prompt taxonomy.** Specifically the `success_analysis` / `failure_analysis` / `factual_content` / `skip_learning` four-way classification with named fields per category (especially failure: failure-point + flawed-reasoning + correct-approach + prevention-principle + applicable-conditions) and the explicit anti-inflation guardrail. Port this into `/lattice:extract-learnings` as a structured router prompt. Source: `src/server/core/acontext_core/llm/prompt/skill_distillation.py`. ~1 hour of prompt-engineering work, zero infrastructure.

**Consider one optional tactic: auto-prompt extract-learnings on session-end.** The Acontext insight that "the moment of agent stop is when distillation is cheapest" is correct. Lattice could implement a `Stop` hook in `.claude/settings.json` that *suggests* (not auto-executes) running `/lattice:extract-learnings` when the session involved a real learning event (commit landed, bug fix, spec implemented). This preserves the "user decides what's durable" policy while reducing the friction of remembering. Implementation: a hook that posts a notification with a one-click `/lattice:extract-learnings` invocation. ~30 min.

**Skip the MCP plugin / hook-based auto-capture entirely.** The "every Stop event silently distills the transcript and edits skill files" model is the wrong default for a project where (a) user opt-in is meaningful signal and (b) auto-mutation of memory files would conflict with the commit-intent protocol (CLAUDE.md rule 23) — autopilot's git operations would sweep up auto-generated memory edits.

Net: Acontext is a well-engineered system targeting a problem (general-purpose agent skill capture for Claude-Code users without a structured project) that Lattice has already solved better at the storage and provenance layers. The capture-pipeline prompts are worth reading and partially porting; nothing else.
