## Performance fix — proposed 5-phase plan (not yet started)

### Phase 1: Split SPA to Render Static Site (CDN-backed, free)
- Deploy React build artifacts as separate Render Static Site
- API server becomes API-only (no static file serving)
- Add VITE_API_BASE_URL env var to frontend build
- Cons: Two services to deploy, need API base URL at build time
- CORS already permissive (`allow_origins=["*"]`)

### Phase 2: Add LRU caching to `analysis_views.py`
- `_load_from_disk()` currently has ZERO in-memory cache (parses JSON from disk every call)
- Add mtime-keyed `@lru_cache` like `analyses.py` already has for unified_findings
- Biggest single improvement for default-settings path
- Cons: ~80-120MB memory for cached dicts (fine on 2GB for 1 study)

### Phase 3: Add GZip middleware
- `from fastapi.middleware.gzip import GZipMiddleware`
- Reduces ~8MB JSON transfer to ~800KB per view load
- Must be done AFTER Phase 2 (otherwise adds CPU to already-slow uncached path)
- Cons: ~100-200ms CPU per large response for compression

### Phase 4: Add uvicorn workers
- `--workers 2` in Dockerfile CMD
- Allows concurrent request processing (frontend fires 4-6 parallel requests per view)
- Cons: ~500MB per worker, 2 workers = ~1GB on 2GB RAM (tight but feasible)
- Cons: Each worker has independent LRU cache
- Less critical after Phase 2 caching (serial queue becomes 6 × ~30ms = ~180ms)

### Phase 5: Bake overrides into generated files (production milestone)
- Apply pattern overrides at generation time, not request time
- Unlocks true static file serving for all views
- Deferred — not needed now

The diagnosis identified 5 compounding factors (single worker, no view caching, no compression, non-default settings pipeline, SPA through Python). The fix is phased: Phase 2 (LRU cache) is highest-impact and lowest-risk. Phase 1 (SPA split) is highest-impact for perceived load time. Phase 3-4 are complementary. Phase 5 is deferred to production.

Key architectural insight: SENDEX serves pre-computed static JSON through a Python app server that parses, processes, and re-serializes it on every request. This is fundamentally wrong for a read-heavy app. The fix trajectory is: cache in memory → compress → parallelize → eventually serve static.
