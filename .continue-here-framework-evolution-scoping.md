## Decisions made this session

- **Typed records over regex tightening.** When the user pushed back on the band-aid regex approach, the merit-driven answer was structural: "an agent (skill / hook / autopilot) makes a load-bearing decision by reading prose written for humans, when the underlying data is small enough to live in typed records." Every fix in this session followed that pattern.
- **Defer the corpus distill until settling moment.** User's instinct from turn 2 was correct — distill IS the right tool for backlog coherence. But running it during the most active phase of the audit produces stale output. Right cadence: after AUDIT-21 + AUDIT-19 ship (closes 13 of 30 SFs).
- **Per-stream + cross-stream meta distill, not whole-corpus.** When run, scope each distill to one Stream's blast radius. Six runs total. Tractable; "distill the whole corpus" is unscoped.
- **Sidecar YAML over front-matter for typed records.** Phases C/D/E all use sidecar files (`escalations.jsonl`, `todo-index.yaml`, `registry-index.yaml`) rather than embedding typed metadata into the human-readable source. Reason: separate file forces a different maintenance discipline mechanically enforced by sync-check hook; front-matter has the same drift risk as inline annotations.
- **Append-only JSONL for ESCALATION.md.** Parallel-session safe via POSIX atomic append; resolutions are events, not in-place edits; state is the fold of events.
- **Default-to-advisory on uncertain migrations.** Phase D + Phase E migrations heuristic-extract from existing prose. Uncertain extractions default to non-blocking and surface for review — preferring false-passing over false-blocking is safer for the migration itself; users can promote items to blocking after seeing them in operation.
- **No "manual --force" steps.** Caught one in my own proposal (lattice→pcc sync fix); user rejected. Replaced with surgical file-level diff + skip-on-conflict + auto-retry on next clean state. Fully automatic; no human in the loop.
- **Dogfood loop is the program's success criterion.** The framework-evolution work itself becomes a feature: each phase, once shipped, lets autopilot advance the next one. Phases A + C + D + E + H + B together close the loop; framework runs itself thereafter.
